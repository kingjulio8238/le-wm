# LeHarness as S1.5: Tactical Planning Between VLM and Motor Policy

## The Problem

The robotics field is converging on a dual-system architecture:

- **S2 (System 2):** A VLM (7-55B params, cloud or high-end GPU, 1-10 Hz) that understands *what* to do — task decomposition, scene understanding, language grounding.
- **S1 (System 1):** A small motor policy (2-50M params, on-robot, 50-1000 Hz) that knows *how* to move — joint torques, end-effector deltas, compliance.

This architecture has two gaps:

1. **No physics-grounded planning between S2 and S1.** The VLM says what to do, the motor policy executes. Nobody checks whether the plan will physically work before trying it.

2. **No feedback from S1 to S2.** If the motor policy can't achieve the goal, the VLM doesn't know until it re-runs and notices the failure visually — which could be seconds later.

LeHarness fills both gaps as an **S1.5 tactical planner**: it imagines thousands of futures per decision, catches bad plans before execution, and signals the VLM when it needs a new goal. At 15M params, it runs on edge hardware.

## How the Industry Does S1/S2 Today

### Interface Patterns Across Major Systems

| System | S2→S1 Signal | Dimensionality | Mechanism | S1→S2 Feedback |
|--------|-------------|----------------|-----------|----------------|
| **GR00T N1** | VLM token sequence | ~64 tokens × 1536-dim | Cross-attention into DiT | None |
| **Figure Helix** | Single latent vector | ~4096-dim (projected) | Concatenation with S1 visual features | None |
| **Fast-in-Slow VLA** | LLM hidden states | 4096-dim | Shared transformer blocks (last 2 of 32) | Implicit (shared params) |
| **OpenHelix** | `<ACT>` token embedding | 4096 → 512 linear projection | Goal conditioning in diffusion policy | None (MLLM frozen) |
| **DreamZero** | N/A (joint model) | Concatenated video + action latents | Shared 14B DiT | KV cache replacement with real obs |

### Key Details Per System

**GR00T N1** (NVIDIA, March 2025):
- S2 is Eagle-2 VLM (later Cosmos-Reason-2B in N1.6). Produces vision-language token embeddings from layer 12 of the VLM's LLM component (layer 12 chosen as speed/quality tradeoff).
- Images encoded into 64 token embeddings per camera frame (via pixel shuffle from 224×224).
- S1 DiT uses alternating self-attention and cross-attention blocks (Flamingo/VIMA style). Cross-attention attends to VLM output tokens.
- S2 runs at 10 Hz on NVIDIA L40. S1 outputs action chunks of 16 timesteps at 120 Hz motor rate.
- Both systems jointly trained end-to-end. In N1.6, top 4 VLM layers are unfrozen during pretraining.

**Figure Helix**:
- S2 is a 7B VLM fine-tuned on 500+ hours of robot demonstrations. Distills all semantic/task-relevant information into a single continuous latent vector.
- S1 is an 80M-parameter cross-attention encoder-decoder transformer at 200 Hz.
- The latent vector is projected into S1's token space and concatenated with visual features from S1's own stereo vision backbone.
- S2 runs as an asynchronous background process at 7-9 Hz. S1 reads the latest S2 latent on each cycle.
- Fully end-to-end: gradients backprop from S1 into S2 via the latent communication vector.

**Fast-in-Slow VLA** (NeurIPS 2025):
- S1 is NOT a separate model. The final 2 transformer blocks of the same LLM are repurposed as S1, while blocks 1-30 serve as S2. S2 outputs intermediate latent features from the split point.
- S1 receives additional high-frequency inputs: 3D point clouds, fresh 2D images, proprioception — tokenized and injected alongside S2's latent features.
- Action generation uses diffusion denoising within the final blocks. S2:S1 frequency ratio is 1:4. S1 achieves 117.7 Hz.
- Dual-aware co-training: diffusion denoising loss + autoregressive language modeling loss applied simultaneously.

**OpenHelix** (open-source):
- S2 is a frozen LLaVA-7B. A learnable `<ACT>` token is appended to the language instruction. The hidden state at the `<ACT>` position is extracted (4096-dim) and projected to 512-dim via a single linear layer.
- S1 is a pre-trained 3D Diffuser Actor. The 512-dim latent serves as a goal conditioning signal within the diffusion policy's denoising process.
- Auxiliary prediction head on the `<ACT>` embedding predicts gripper state, 3D end-effector position, and 6D rotation — forces the MLLM to attend to visual information, not just parrot text.
- Most modular design: MLLM is frozen, only `<ACT>` embedding + projection are trained.

**DreamZero** (NVIDIA, Feb 2026):
- 14B autoregressive DiT that jointly denoises video latents and action latents in the same forward pass.
- Video and action latents concatenated along the sequence dimension, processed by shared transformer backbone with separate encoder/decoder heads per modality.
- DreamZero-Flash: decoupled noise schedule — actions denoised to clean in a single step (38× speedup).
- Inference at 7 Hz on GB200. Action chunks of 48 steps at 30 Hz control frequency.
- Ground-truth observations replace predicted video in KV cache after each chunk — the functional equivalent of reality-checking the world model.

### Critical Observation

**Nobody has S1→S2 feedback.** In every system, the flow is strictly S2→S1. S2 re-runs at its cadence on the latest camera frame, with no signal from S1 about whether the plan is working. This is the gap LeHarness can fill.

## What VLMs Actually Output

### Action/Goal Representations Across Systems

| System | Visual Encoder | Language Encoder | Output Format |
|--------|---------------|-----------------|---------------|
| **Pi0/Pi0.5** | SigLIP (in PaliGemma) | Gemma (in PaliGemma) | Continuous action chunks via flow matching (H=50 steps) |
| **RT-2** | PaLI-X / PaLM-E | Same model's text encoder | 256-bin discretized action tokens as text ("1 128 91 241...") |
| **OpenVLA** | SigLIP + DINOv2 fused | Llama-2 | 256-bin discretized action tokens (7 token IDs) |
| **Octo** | Lightweight CNN patchifier | T5-Base (frozen, 16 tokens × 768-dim) | Continuous action chunks via diffusion MLP head |
| **GR00T N1** | Eagle VLM visual encoder | SmolLM-1.7B | Continuous actions via DiT flow matching |
| **Cosmos Reason** | Video understanding | Chain-of-thought text | Natural language reasoning (not motor commands) |

### Three Tiers of Goal Specification

**Tier 1: Dense latent embeddings**
- Used by: Octo (T5 embeddings → transformer → readout), GR00T N1 (VLM latents → DiT), Pi0 (VLM tokens → action expert via cross-attention in 4608-dim shared space)
- Format: Dense float vectors in a learned embedding space
- Pro: Maximally expressive, differentiable, low-latency
- Con: Not interoperable across models without alignment

**Tier 2: Discrete action/goal tokens**
- Used by: RT-2, OpenVLA, Pi0-FAST
- Format: Integer token IDs from a shared or dedicated vocabulary
- Pro: Plugs directly into LLM autoregressive generation
- Con: Quantization loss, only works when the goal IS an action sequence

**Tier 3: Structured natural language text**
- Used by: SayCan, Code-as-Policies, Inner Monologue, Cosmos Reason
- Format: Text strings (skill names, Python code, chain-of-thought reasoning)
- Pro: Maximally interoperable, human-readable, any LLM can produce it
- Con: Requires parsing/grounding, lossy for spatial precision

### Key Takeaway for Goal Interface

The projection layer is the adapter. You need one per upstream VLM family. LeHarness already has this pattern with `CLIPProjection` and `CoordProjection` — adding adapters for SigLIP, T5, Eagle, or PaliGemma embeddings is the same mechanism: train a small MLP (N-dim → 192-dim) on paired (goal_image, VLM_embedding) data.

### What the World Model Buys You

**1. Physics-grounded lookahead that neither S1 nor S2 has.**
S2 can say "reach from the left" based on semantic reasoning. The world model can *simulate* reaching from the left vs. right and pick the one that works. This is valuable for contact-rich manipulation where the right approach angle isn't obvious from semantics alone.

**2. Failure detection before execution.**
The model dreams a plan and checks: "does this trajectory end near the goal?" If not, don't execute — replan or escalate to S2. Reactive S1 policies can't provide this.

**3. Edge-local deliberation.**
At 15M params, you can "think" on the robot without a cloud round-trip. The VLM sets the goal once; the world model handles tactical replanning locally at 3-10 Hz.

**4. Goal-space bridge.**
Language conditioning (N2) means S2 can pass goals as text → the world model converts to its 192-dim planning space → plans → S1 executes. The world model is the translation layer between VLM-speak and motor-speak.

**5. Closed-loop S1→S2 feedback.**
This is genuinely novel. Every other dual-system architecture is open-loop from S2's perspective. LeHarness can close that loop by reporting confidence, predicted outcomes, and drift back to the VLM.

### Honest Tensions

**Better S1 policies might make the world model unnecessary.** Diffusion policies and action chunking transformers are getting very good at handling multimodal, contact-rich manipulation reactively. If S1 is good enough to handle the tactical layer without lookahead, the world model is overhead. Counterargument: reactive policies fail on novel situations outside their training distribution. The world model can search for solutions the policy was never trained on.

**VLMs are getting faster.** Pi0 runs at ~50 Hz. If S2 gets fast enough to close the loop itself, the middle layer disappears. Counterargument: that's still a 3B+ model requiring serious GPU. 15M on a Jetson is a fundamentally different cost/power profile.

**The world model only knows what it's trained on.** It can't plan for a robot arm if it's only seen 2D navigation. Task-specific weights are the bottleneck. This means retraining for every new embodiment/task — which undermines the "general tactical planner" story. Counterargument: the planning stack (CEM, Dream Tree, language conditioning) transfers; only the dynamics model needs retraining.

## The Concrete Interface Specification

### Inbound: S2 → LeHarness (Goal Adapter)

```python
class GoalAdapter:
    """Maps any upstream VLM goal format to 192-dim planning space."""

    # Already works today:
    def from_image(self, img: np.ndarray) -> Tensor:
        """(H,W,3) uint8 → preprocess → ViT encode → (1,1,192)"""

    def from_text(self, text: str) -> Tensor:
        """String → CLIP/coord encode → MLP project → (1,1,192)"""

    # New — direct embedding injection from VLMs:
    def from_vlm_embedding(self, emb: Tensor, source: str) -> Tensor:
        """
        Raw VLM embedding → trained projection → (1,1,192).

        source: "siglip" | "clip" | "t5" | "eagle" | "paligemma"
        Each has a trained projection MLP: N-dim → 192-dim.
        Trained on paired (goal_image, VLM_embedding) data
        with MSE loss between projection(VLM_emb) and ViT_encode(goal_image).
        """

    # New — subgoal sequences for multi-step tasks:
    def from_subgoals(self, goals: list) -> list[Tensor]:
        """
        Ordered list of goals (any format) → list of (1,1,192) embeddings.
        LeHarness plans toward goals[0], detects arrival (cost drops
        below threshold), advances to goals[1], etc.

        This is VLM-provided subgoal sequencing — unlike Dream Chaining (D2)
        which failed because latent interpolation produces unreachable states,
        these subgoals come from the VLM's semantic understanding of the task.
        """
```

**Projection adapters needed per VLM family:**

| VLM Family | Source Dim | Projection | Training Data |
|-----------|-----------|------------|---------------|
| CLIP ViT-B/32 | 512 | MLP → 192 | Already trained (N2) |
| SigLIP (Pi0/OpenVLA) | 768 | MLP → 192 | (goal_image, SigLIP_emb) pairs |
| T5-Base (Octo) | 768 | MLP → 192 | (goal_image, T5_emb) pairs |
| Eagle/SmolLM (GR00T) | 1536 | MLP → 192 | (goal_image, Eagle_emb) pairs |
| PaliGemma (Pi0) | 4608 | MLP → 192 | (goal_image, PaliGemma_emb) pairs |

Training recipe is identical for all: freeze the VLM encoder, freeze the LeWM ViT encoder, train only the projection MLP to minimize MSE between `projection(VLM_encode(goal_image))` and `ViT_encode(goal_image)`. Same approach as existing `CLIPProjection` and `CoordProjection`.

### Outbound: LeHarness → S2 (Confidence Signal)

```python
@dataclass
class PlanResult:
    """Returned by every planning call. Action goes to S1, signals go to S2."""

    # To S1 motor policy:
    action: np.ndarray              # (action_dim,) — the planned action

    # To S2 VLM (confidence signals):
    confidence: float               # 0.0-1.0, derived from normalized CEM cost
    planning_cost: float            # Raw MSE cost of best CEM trajectory
    terminal_embedding: Tensor      # (1,1,192) — where the plan predicts ending up
    planability: float              # _score_state() result — can we keep planning from there?

    # Optional (requires DreamScorer):
    value_estimate: float           # Learned value V(z_terminal, z_goal)
    uncertainty: float              # Ensemble disagreement at terminal state

    # Derived signals:
    @property
    def needs_replan(self) -> bool:
        """High cost + low planability → ask S2 for a new goal."""
        return self.confidence < self.replan_threshold

    @property
    def goal_unreachable(self) -> bool:
        """Planning cost hasn't improved over last N steps."""
        return self._cost_trend_flat()
```

**Where the confidence signals come from (all exist in current code):**

| Signal | Source | What It Means |
|--------|--------|---------------|
| `planning_cost` | `_evaluate_candidates()` topk_vals | MSE between best trajectory terminal and goal. High = goal is far/hard. |
| `planability` | `_score_state(terminal_emb, goal_emb)` | How easy is it to keep planning from the predicted future state. Low = dead end. |
| `confidence` | Normalized inverse of planning_cost | 0-1 scalar for simple thresholding. |
| `terminal_embedding` | `_cem_plan(return_terminal_emb=True)` | The predicted 192-dim state after executing the plan. Enables drift detection. |
| `value_estimate` | `DreamScorer.ensemble.predict()` | Learned value V(z_terminal, z_goal) ∈ [0,1]. |
| `uncertainty` | `DreamScorer.ensemble.predict_with_uncertainty()` | Std across 5 ensemble members. High = unfamiliar territory. |

### Drift Detection Loop

After each action execution, compare prediction vs. reality:

```python
class DriftDetector:
    """Monitors prediction accuracy to trigger S2 replanning."""

    def __init__(self, threshold: float = 0.1, window: int = 5):
        self.threshold = threshold
        self.window = window
        self.history = []

    def check(self, predicted_terminal: Tensor,
              actual_observation: Tensor,
              pipeline: PlanningPipeline) -> DriftSignal:
        """
        Compare where we predicted we'd be vs. where we actually are.

        predicted_terminal: (1,1,192) from last PlanResult.terminal_embedding
        actual_observation: (H,W,3) current camera frame → encode → (1,1,192)
        """
        actual_emb = pipeline.encode(pipeline.preprocess(actual_observation))
        drift = ((predicted_terminal - actual_emb) ** 2).sum().item()
        self.history.append(drift)

        return DriftSignal(
            drift_mse=drift,
            drift_exceeded=drift > self.threshold,
            trend_increasing=self._trend_increasing(),
            # If drift is large and increasing, the world model
            # is systematically wrong → something unexpected happened
            # → escalate to S2 for replanning
            escalate_to_s2=drift > self.threshold and self._trend_increasing()
        )
```

This is cheap: one encoder call per step (already happening) plus a vector comparison. The encoder runs in ~4ms.

### The S2 Replan Trigger

```python
# In the main control loop:
while not done:
    # S1.5: Plan
    result = pipeline.plan(obs)  # Returns PlanResult

    # Check confidence
    if result.needs_replan:
        # Tell VLM: "I can't make progress toward your goal"
        new_goal = vlm.replan(
            obs=obs,
            original_goal=current_goal,
            failure_reason="low_confidence",
            planning_cost=result.planning_cost,
            terminal_prediction=result.terminal_embedding
        )
        pipeline.set_goal(new_goal)
        continue

    # S1: Execute
    motor_policy.execute(result.action)

    # Get new observation
    new_obs = camera.capture()

    # Check drift
    drift = drift_detector.check(
        predicted_terminal=result.terminal_embedding,
        actual_observation=new_obs,
        pipeline=pipeline
    )
    if drift.escalate_to_s2:
        new_goal = vlm.replan(
            obs=new_obs,
            original_goal=current_goal,
            failure_reason="drift_detected",
            drift_mse=drift.drift_mse
        )
        pipeline.set_goal(new_goal)

    obs = new_obs
```

## The World Model Debate: Context

### Who's Betting on World Models (2025-2026)

- **Yann LeCun** left Meta, raised $1.03B for AMI Labs (March 2026, $3.5B valuation) to build world models via JEPA. LeWM is from his research lineage.
- **NVIDIA's Jim Fan** declared 2026 "the foundational year for world models." DreamZero (14B World Action Model) showed 2× improvement over VLAs in generalization with zero-shot task completion.
- **Fei-Fei Li** raised $1B for World Labs ($5.4B valuation).
- **Skild AI** — general-purpose robotic brain using Cosmos WFMs, SoftBank/NVIDIA in talks at $14B valuation.
- **1X** — first to forecast contact-rich futures for full-body humanoid, uses world model for policy evaluation.
- **V-JEPA 2** (Meta) — achieved 80% zero-shot pick-and-place on real Franka arms with 62 hours of unlabeled video, zero task-specific training.

### Who's Betting on VLAs

- **Physical Intelligence** — Pi*0.6 folds laundry in real homes, assembles boxes in factories, makes espresso autonomously. No world model needed.
- **Google DeepMind** — RT-2/RT-2-X, treating actions as text tokens from a VLM.
- **Stanford** — OpenVLA, open-source VLA following the RT-2 paradigm.

### The Emerging Consensus

The "replacement" framing is wrong. The field is converging on **integration**:

- VLAs provide semantic understanding and language grounding
- World models provide physics simulation and future prediction
- The dual-system architecture combines both
- The key open question is where the world model sits: as runtime planner (S1.5), as training infrastructure (offline data generation), or as evaluation tool (policy scoring in imagination)

LeHarness's position: **runtime S1.5 tactical planner**, made possible by the 15M-param efficiency that larger world models (DreamZero at 14B, Cosmos at billions) cannot match on edge hardware.

## Next Steps

Based on the on-pod validation results, these are the recommended next steps in priority order.

### N1: Default to confidence-only replanning (disable drift detection)

The drift sweep data is unambiguous: every drift replan on TwoRoom makes things worse. At threshold=750 with zero drift replans, S1.5 hits 75% vs 65% baseline — a 10pp improvement from confidence replanning alone. At threshold=100-200, drift replans collapse success to 50-55%.

The root cause: drift MSE on TwoRoom averages ~290-314 across 192 dimensions. Any threshold low enough to trigger drift detection also triggers on normal planning variance. The signal-to-noise ratio is too low for this task.

**What to do:** Change the default `drift_threshold` to `float('inf')` (effectively disabled). Keep drift detection available as an opt-in for tasks with higher dynamics variance. Document that confidence replanning is the primary feedback mechanism.

**Off-pod:** One-line default change in `S15ControlLoop.__init__()`, update docs. Update unit tests if needed.

**On-pod:** Re-run `eval_s15_integration.py` with `--drift-threshold 999999 --num-eval 50` to get the headline confidence-only result at scale.

### N2: Smarter replan strategy in SimVLM

Currently, when confidence is low and the loop replans, `SimVLM` provides an alternative goal from a nearby dataset state (5 steps earlier in the same episode). But if the planner can't reach the original goal, a goal from 5 steps earlier is still roughly the same location — the replan doesn't actually help.

Better strategies to implement and compare:
- **Intermediate waypoint:** If goal is far (high planning_cost), encode an observation from 50% of the way to the goal as an intermediate subgoal. Plan toward that instead.
- **Skip and execute:** If confidence is low but not catastrophic (e.g., 0.03-0.05), execute the action anyway rather than wasting a planning step replanning toward the same goal.
- **Adaptive cost_scale:** Track running statistics of planning_cost and normalize confidence relative to recent history, not a fixed scale.

**Off-pod:** Implement new replan strategies in `SimVLM` (add strategy parameter). Write unit tests with `LightweightPipeline`.

**On-pod:** Run `eval_s15_integration.py` with each strategy, compare success rates. The 50-episode eval is the benchmark.

### N3: Integrate S1.5 confidence replanning with Dream Tree

Dream Tree (D3) improves flat CEM by 35-62% through structured exploration. S1.5 confidence replanning adds another layer of feedback. These haven't been tested together. When the tree planner can't find a good action across all 4 roots, confidence should be low — triggering a replan that adjusts the goal rather than continuing to search in a dead end.

**Off-pod:** Modify `eval_s15_integration.py` to optionally create a `DreamTreePlanner` wrapping the pipeline. The tree planner's `plan()` returns `np.ndarray` (not `PlanResult`), so either: (a) wrap it to produce `PlanResult`, or (b) run the tree inside the pipeline before the confidence check.

**On-pod:** Run the modified eval with `--tree` flag. Compare four conditions: flat CEM, flat CEM + S1.5, Dream Tree, Dream Tree + S1.5.

### N4: Unblock Cube dataset for 3D manipulation validation

The S1.5 story — "physics-grounded tactical planner between VLM and motor policy" — needs 3D manipulation data to be compelling. TwoRoom is 2D navigation where the dynamics are simple and drift detection doesn't add much. Contact-rich tasks (Cube) would stress the world model's predictions more, likely producing meaningful drift events where the world model is genuinely surprised by contact dynamics.

**Off-pod:** Contact the paper authors (Lucas Maes et al.) for an alternative download link or HuggingFace mirror. Try `gdown` with explicit OAuth. Check if the OGBench Cube dataset is available from other sources.

**On-pod:** Once data is available, run `eval_dream_tree.py --config-name cube --mode both` to verify the planning stack transfers, then run `eval_s15_integration.py --config-name cube` for S1.5 validation on a harder task.

### N5: Validate VLM projection adapters with real VLM embeddings

The `GoalAdapter` and `VLMProjection` code is built and unit-tested, but no projection has been trained on real VLM embeddings yet (only CLIP and coordinate projections exist from N2). Training a SigLIP projection would validate the full S2→S1.5 goal pipeline.

**Off-pod:** Write a data collection script that takes goal images from the TwoRoom dataset, encodes them through both the LeWM ViT encoder and a SigLIP encoder, and saves the paired embeddings. The `train_vlm_projection.py` script already handles the training.

### Step 2: Confidence-Only S1.5 at Scale (N1 — headline number)

The drift sweep from the first on-pod session (2026-03-28) showed confidence-only replanning outperforms baseline (75% vs 65% at threshold=750, 20 episodes). Drift detection hurt on TwoRoom. The code now defaults to confidence-only (`drift_threshold=inf`).

Run 50 episodes with confidence-only to get the headline number.

```bash
python scripts/eval_s15_integration.py \
    --policy tworoom/lewm \
    --config-name tworoom \
    --num-eval 50 \
    --eval-budget 100 \
    --cost-scale 200.0 \
    --replan-threshold 0.05 \
    --seed 42 \
    --output results/s15_confidence_only_50ep.json
```

**What success looks like:** S1.5 success rate >= baseline. Confidence replans > 0, drift replans = 0. Mean confidence in [0.3, 0.9].

**Calibration notes (from 2026-03-28 session):**
- Planning costs on TwoRoom average ~40-75 MSE. `cost_scale=200` maps this to confidence ~0.6-0.8.
- The pipeline default `cost_scale=10.0` is too low for TwoRoom — always pass `--cost-scale 200` or higher.
- `replan_threshold=0.05` triggers replanning only for genuinely hopeless plans.

### Step 3: Compare Replan Strategies (N2)

Three strategies are now available. Compare them on the same episodes to find which replan response works best.

```bash
for strat in waypoint nearby persist; do
    echo "=== strategy=$strat ==="
    python scripts/eval_s15_integration.py \
        --policy tworoom/lewm \
        --config-name tworoom \
        --num-eval 50 \
        --cost-scale 200.0 \
        --replan-threshold 0.05 \
        --replan-strategy $strat \
        --seed 42 \
        --output results/s15_strategy_${strat}.json
done
```

**What success looks like:** At least one strategy beats baseline. Compare success rates:
- **waypoint** — provides a midpoint subgoal (should help when goal is far)
- **nearby** — provides an earlier dataset state as goal (original behavior)
- **persist** — returns same goal, lets planner keep trying (skip-and-execute)

If `persist` performs best, replanning the goal doesn't help — the confidence signal's main value is just the extra planning step (CEM re-randomizes).

## Failure Modes & What They Mean

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| Baseline <30% on TwoRoom | Wrong checkpoint or env config | Verify `tworoom/lewm` exists at `$STABLEWM_HOME/tworoom/lewm/lewm_object.ckpt` |
| S1.5 success rate << baseline | Too many false replans | Raise `--drift-threshold` (try 500-750) and/or lower `--replan-threshold` (try 0.05) |
| Zero confidence replans | Confidence always high | Lower `--cost-scale` (try 100-150) |
| Zero drift replans | Drift threshold too generous | Lower `--drift-threshold` (try 350-500). Drift MSE on TwoRoom averages ~290 |
| Confidence always ~0 | `cost_scale` too low | Raise `--cost-scale` (try 200-300). Planning costs on TwoRoom average ~40-75 |
| Confidence always ~1 | `cost_scale` too high | Lower `--cost-scale` (try 100) |
| `AssertionError: _autoreset_envs` | Gym autoreset after terminated/truncated | Fixed in current code — `git pull` |
| Script crashes on import | Missing dependencies | Run `pip install "stable-worldmodel[train,env]"` |
| `KeyError: 'pixels'` | Wrong dataset format | Check HDF5 keys: `python -c "import h5py; f=h5py.File('path.h5','r'); print(list(f.keys()))"` |
| `MUJOCO_GL` error | Rendering backend | `export MUJOCO_GL=egl` (or `osmesa` on CPU-only) |

### API Quick Reference

```python
# --- Core planning (returns PlanResult) ---
from harness.pipeline import PlanningPipeline

pipeline = PlanningPipeline("tworoom/lewm")
pipeline.warmup()
result = pipeline.plan(obs_image, goal_image)
result.action             # (10,) numpy — send to motor
result.confidence         # 0.0-1.0
result.needs_replan       # True if confidence < threshold
result.terminal_embedding # (1,1,192) — for drift detection

# --- Goal injection from VLMs ---
from harness.goal_adapter import GoalAdapter

adapter = GoalAdapter(pipeline)
adapter.load_projection("siglip", "path/to/weights.pt")
goal_emb = adapter.from_vlm_embedding(vlm_output, source="siglip")
pipeline.set_goal_embedding(goal_emb)

# --- Subgoal sequencing ---
from harness.subgoal_sequencer import SubgoalSequencer

subgoals = [emb1, emb2, emb3]
seq = SubgoalSequencer(pipeline, subgoals, arrival_threshold=1.0)
while not seq.is_complete:
    result = pipeline.plan(obs)
    seq.step(result)

# --- Drift detection ---
from harness.drift_detector import DriftDetector

detector = DriftDetector(threshold=0.5, window=5)
signal = detector.check(predicted=result.terminal_embedding, actual_obs=new_obs, pipeline=pipeline)
if signal.escalate_to_s2:
    vlm.replan(...)

# --- Full S1.5 loop (zero mocks, on-pod) ---
from harness.s15_loop import S15ControlLoop
from harness.sim_components import SimVLM, SimMotorPolicy

vlm = SimVLM(pipeline, goal_image, dataset=dataset, episode_indices=ep_idx, goal_step=30)
motor = SimMotorPolicy(world, process, pipeline._action_dim)
loop = S15ControlLoop(pipeline, vlm, motor, drift_threshold=0.5)
stats = loop.run_episode(initial_obs=obs, max_steps=100)
```
